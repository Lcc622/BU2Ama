## 2. 标题颜色替换残留修复

### 问题
加色时替换标题中的颜色名，源标题 "Blush Pink" 中 "Blush" 被替换为新颜色名，但 "Pink" 作为独立词残留。例如：`...Formal Dresses Light Green Floral Pink US4`。

### 根因
`replace_color_whole_words()` 仅匹配映射中的颜色名（如 "Blush"），不处理标题中的组合颜色短语（如 "Blush Pink"）。

### 修复
**`backend/app/core/excel_processor.py`**：

扩展替换正则，匹配源颜色名 + 可选的尾随颜色关键词：

```python
# 常见颜色关键词集合（pink, blue, green, gold 等）
_trailing_color_alt = "|".join(re.escape(kw) for kw in _color_keywords)

# 原来：只匹配 "Blush"
# 现在：匹配 "Blush" 或 "Blush Pink"（后接颜色词时一起替换）
replaced = re.sub(
    rf'(?<![A-Za-z0-9]){re.escape(source_text)}(?:\s+(?:{_trailing_color_alt}))?(?![A-Za-z0-9])',
    target_text, text, flags=re.IGNORECASE,
)
```

### 效果
- "Blush Pink" → "Light Green Floral"（整体替换，无残留）✅
- "Blush" 后接非颜色词 → 只替换 "Blush"（不误删）✅
- Generic Keyword 同样修复 ✅

## 3. Color Map 分类逻辑修复

### 问题
"Light Green Floral" 的 Color Map 应为 "Multicoloured"，但 `get_color_map_value()` 先匹配到 "green" 关键词，返回 "Green"。而且 "floral" 根本不在 Multicolor 关键词列表中。

### 修复
**`backend/app/core/excel_processor.py`** — `get_color_map_value()`：

1. 新增 `floral`、`tie dye`、`rainbow` 等关键词到多色分类
2. **Multicoloured 优先匹配**：含 floral/print/pattern 的颜色名直接返回 "Multicoloured"，不进入单色分类
3. 统一分类名为 "Multicoloured"（原来是 "Multicolor"）

```python
# Multicolor 关键词优先匹配
multicolor_keywords = ['multicolor', 'multi', 'floral', 'print', 'pattern', 'tie dye', 'tie-dye', 'rainbow']
if any(keyword in color_lower for keyword in multicolor_keywords):
    return 'Multicoloured'
```

### 效果
- `Light Green Floral` → `Multicoloured` ✅
- `Green` → `Green` ✅（普通绿色不受影响）
- `Tie Dye` → `Multicoloured` ✅

## 4. 店铺代码完全拆分（EP / DM·PZ 独立项目）

### 背景
3/30 的双实例方案虽然实现了数据隔离，但代码层仍然共享：改 EP 逻辑可能影响 DM/PZ。为彻底消除串店风险，将代码完全拆分。

### 方案
同一 repo，两个独立子项目：

```
BU2Ama/
├── ep/                     # EP 完全独立项目
│   ├── backend/            # 只含 EPUS 逻辑
│   ├── frontend/           # 默认 EPUS 模板
│   └── docker-compose.yml  # backend:8001 + frontend:5173
│
├── dmpz/                   # DM/PZ 完全独立项目
│   ├── backend/            # 只含 DaMaUS + PZUS 逻辑
│   ├── frontend/           # 默认 DaMaUS 模板
│   └── docker-compose.yml  # backend:8003 + frontend:5174
│
├── backend/                # 旧共享代码（保留备份）
├── frontend/               # 旧共享前端（保留备份）
└── docker-compose.yml      # 旧配置（保留备份）
```

### 核心改动

**EP 后端**（`ep/backend/app/config.py`）：
- 移除 `STORE_GROUP` 动态检测，硬编码 EPUS 配置
- `TEMPLATES = {"EPUS": {...}}`，`ALLOWED_STORE_PREFIXES = ("EP",)`

**DM/PZ 后端**（`dmpz/backend/app/config.py`）：
- 硬编码 DaMaUS + PZUS 配置
- `TEMPLATES = {"DaMaUS": {...}, "PZUS": {...}}`

**EP excel_processor.py**：
- 删除所有 `if template_type in {"DaMaUS", "PZUS"}` 分支
- 删除 DM/PZ 专属逻辑（restock_date、product_tax_code、embellishment、quantity=0 等）

**DM/PZ excel_processor.py**：
- 删除 EPUS 专属逻辑（加色模板选择、quantity=5 等）
- 保留 DM/PZ 特有逻辑（force variation theme、embellishment 处理等）

**前端**：各自独立一份，模板选择器只显示本店模板

**docker-compose.yml**：各自独立

### 隔离验证
```
EP 请求 DaMaUS 模板 → "当前实例不支持模板类型: DaMaUS" ✅
DM/PZ 请求 EPUS 模板 → "当前实例不支持模板类型: EPUS" ✅
EP 加色 GF → Color=Light Green Floral, Map=Multicoloured ✅
```

## 5. 跟卖属性字段串值修复

### 问题
跟卖导出 EE01962 时，Fabric Type 显示 "Sequin" 而非正确的 "Chiffon"，Outer Material Type、Sleeve Type 等也可能串值。

### 根因
跟卖模式下，`Fabric Type` 和 `Outer Material Type` 的 `read_source_value` 使用了 `source_header_map` / `source_row_values`（当前匹配的源行），而非 `canonical_*`（同款同色第一条记录）。当同一款号有多条源记录时，可能读取到错误源行的属性值。

而 `Item Length Description` 已经正确使用了 `display_source_*`。

### 修复
**`ep/backend/app/core/excel_processor.py`**：

1. 跟卖模式下，将 `display_source_*` 指向 `canonical_*`：
```python
if follow_sell_mode:
    display_source_row_values = canonical_row_values
    display_source_header_map = canonical_header_map
```

2. Outer Material Type 和 Fabric Type 改为使用 `display_source_*`：
```python
material_value = read_source_value(
    display_source_header_map,  # 原来是 source_header_map
    display_source_row_values,  # 原来是 source_row_values
    first_material_candidates,
)
```

### 效果
Fabric Type、Outer Material Type、Item Length Description 统一从 canonical 源读取，不再串值。

## 运维注意事项

- **启动命令变更**：不再使用根目录 `docker-compose up`，改为分别启动：
  - `cd ep && docker-compose up -d`（EP 店铺）
  - `cd dmpz && docker-compose up -d`（DM/PZ 店铺）
- **索引数据库**：ep/ 和 dmpz/ 各自有独立的 `excel_index_*.db`，首次启动需要重新上传源文件建立索引
- **colorMapping.json**：EP 和 DM/PZ 各自维护独立副本（`ep/backend/data/` 和 `dmpz/backend/data/`），新增颜色需要两边都加
- **旧目录保留**：根目录的 `backend/`、`frontend/`、`docker-compose.yml` 暂时保留作为备份，确认无误后可删除
- **代码修复同步**：如果两个店铺都需要的 bug fix，需要在两个目录分别修改（这是拆分的代价）
